## 2. Политика логирования

### Что логируется

| Данные | Логируется? | Уровень | Хранение |
|---|---|---|---|
| Текст запроса пользователя | ✅ Да | INFO | Локальные логи, ротация 7 дней |
| Structured intent (после парсинга) | ✅ Да | INFO | Локальные логи |
| LLM prompts (system + user) | ✅ Да | DEBUG | Локальные логи, только в dev-режиме |
| LLM responses | ✅ Да | DEBUG | Локальные логи, только в dev-режиме |
| Результаты поиска товаров | ✅ Да | INFO | Локальные логи |
| Финальная рекомендация | ✅ Да | INFO | Локальные логи |
| Ошибки и exceptions | ✅ Да | ERROR | Локальные логи + stderr |
| Метрики latency по шагам | ✅ Да | INFO | Локальные логи |

## 3. Работа с персональными данными

### Текущий статус (PoC)

- Система **не требует регистрации** и **не собирает** персональные данные.
- Запросы пользователя **не содержат PII** по дизайну (это товарные запросы).
- Если пользователь случайно введёт PII (e.g. «доставка на ул. Ленина 5, кв. 12»), данные:
  - Не сохраняются в persistent storage.
  - Могут попасть в логи → логи хранятся только локально и ротируются.
  - Передаются в DeepSeek API → подчиняются [DeepSeek Privacy Policy](https://www.deepseek.com/privacy).

- Персональные данные пользователя (имя, email, адрес) — **система их не собирает**.
- API-ключи — **никогда** не попадают в логи (маскирование).
- Платёжные данные — **система не работает** с платежами.

## 4. Защита от Prompt Injection

### Многоуровневая защита
Layer 1: Input Validation
│ - Max length (1000 chars)
│ - Regex-фильтр на известные injection-паттерны
│ ("ignore previous", "system:", "you are now", etc.)
│ - Отклонение запросов с подозрительным содержимым
│
Layer 2: Prompt Architecture
│ - System prompt — hardcoded, не конкатенируется с user input
│ - User input передаётся как отдельное поле (role: user)
│ - Structured output mode (JSON) — LLM отвечает по схеме
│
Layer 3: Output Validation
│ - Pydantic-валидация ответа LLM
│ - Если ответ не парсится в ожидаемую схему → retry или fallback
│ - Проверка: ответ относится к домену (товары) или нет
│
Layer 4: Intent Guard (guard node в LangGraph)
- Regex-based guard node (без LLM-вызова): проверка наличия продуктовых ключевых слов и off-topic паттернов. Не требует обращения к LLM, работает мгновенно.
- Если off-topic → вежливый отказ, pipeline не запускается

## 5. Подтверждение действий (Human-in-the-Loop)

В текущем PoC все действия агента — **read-only** (поиск и анализ), поэтому подтверждение не критично. Однако реализованы следующие принципы:

| Действие | Требует подтверждения? | Реализация |
|---|---|---|
| Уточняющий вопрос | Нет | Автоматически задаётся |
| Поиск товаров | Нет | Read-only операция |
| Анализ отзывов | Нет | Read-only операция |
| Финальная рекомендация | Нет | Показывается с disclaimer |
| Переход на страницу товара (URL) | **Да** | Пользователь сам решает, переходить или нет |
| Покупка | **N/A** | Out of scope — система не покупает |
