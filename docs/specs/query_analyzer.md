# QueryAnalyzer

## 1. Общее описание

Модуль парсинга и уточнения пользовательского запроса. Извлекает структурированный `QuerySpec` из текста на естественном языке. Поддерживает многошаговый диалог уточнения (multi-turn clarification).

### Примеры работы

| Вход (raw query) | Выход (QuerySpec) |
|------------------|-------------------|
| "Хочу беспроводные наушники с шумоподавлением до 15000₽ для бега" | `category: "наушники"`, `budget_max: 15000`, `must_have: ["беспроводные", "шумоподавление", "для бега"]` |
| "ищу ноут для работы" | `category: "ноутбук"`, `must_have: ["для работы"]`, `needs_clarification: true` |
| "телефон" | `category: "телефон"`, `needs_clarification: true` |

---

## 2. Интерфейс

### 2.1 Сигнатура

```python
run_query_analyzer(
    state: dict,
    llm: LLMClient,
    prompts_dir: Path,
    model_override: str | None = None,
) -> dict
```

### 2.2 Входные данные

| Поле | Тип | Описание |
|------|-----|----------|
| user_query | str | Текст от пользователя |
| chat_history | list[dict] | Полная история диалога (для multi-turn) |

### 2.3 Выходные данные

| Поле | Тип | Описание |
|------|-----|----------|
| query_spec | QuerySpec | Структурированный запрос |
| stage | WorkflowStage | Стадия пайплайна (`QUERY_PARSED`) |
| errors | list[str] | Ошибки, если были |
| llm_calls | int | Количество LLM-вызовов |
| total_tokens | int | Суммарное количество токенов |
| total_cost | float | Суммарная стоимость вызовов (USD) |

## 3. Multi-turn Clarification

Когда агент нуждается в дополнительной информации (`needs_clarification=True`), система сохраняет полную историю диалога в `chat_history`. На следующем шаге функция `_build_messages()` формирует цепочку сообщений для LLM:

```
[system_prompt, user_1, assistant_1, user_2, assistant_2, ..., user_N]
```

Это позволяет LLM синтезировать полный запрос из инкрементальных ответов пользователя.

**Пример диалога:**

| Шаг | Роль | Сообщение |
|-----|------|-----------|
| 1 | user | "хочу наушники" |
| 1 | assistant | "Уточните бюджет и назначение (для бега, офиса, музыки?)" |
| 2 | user | "до 15 тысяч, для бега" |
| 2 | assistant | `category: "наушники"`, `budget_max: 15000`, `must_have: ["для бега"]` |

**Логика `_build_messages()`:**
- Всегда первым идёт `system` промпт
- Из `chat_history` берутся только сообщения с ролями `user` / `assistant` (системные пропускаются)
- Если текущий `user_query` не совпадает с последним сообщением в истории — он добавляется явно

## 4. Input Sanitization и защита от Prompt Injection

### 4.1 Ограничение длины

Входной текст обрезается до **1000 символов** перед любой обработкой.

### 4.2 Детекция инъекций

Модуль содержит **16 regex-паттернов**, покрывающих попытки инъекций на английском и русском языках:

**Английские паттерны:**
- `ignore previous instructions` / `ignore all above`
- `disregard previous` / `forget previous`
- `you are now` / `new system prompt`
- `system:` (в начале строки) / `system prompt`
- `act as if you are` / `pretend you are`
- `override instructions` / `<system>` теги

**Русские паттерны:**
- `игнорируй предыдущие`
- `забудь инструкции`
- `ты теперь`
- `новая роль`

### 4.3 Реакция на инъекцию

При обнаружении инъекции вызывается `PromptInjectionError`. Запрос **блокируется до обращения к LLM** — возвращается `QuerySpec` с `needs_clarification=True` и сообщением об ошибке. В `errors` записывается `"Prompt injection detected"`.

---

## 5. Обработка ошибок

| Ситуация | Действие |
|----------|----------|
| Запрос слишком короткий (<2 символов) | Возвращает `needs_clarification=True` с просьбой описать товар подробнее |
| Обнаружена инъекция | Возвращает ошибку + clarification, запрос не передаётся LLM |
| LLM не смог распарсить запрос (exception) | Возвращает `needs_clarification=True` с просьбой переформулировать, ошибка логируется |
| LLM вернул невалидный JSON | Обрабатывается в `LLMClient.call_json()` — автоматический retry с инструкцией вернуть чистый JSON |
